## 0. 문서 간 관계 (중복 방지 목적 명시)

- **인증 방식(4종)은 3차 문서의 "비회원 인증 4종"과 100% 동일** — 재정의 아님, 재확인 목적으로만 기록.
- **정보 수집 자체는 2차 문서의 "자연어 상담 플로우"와 동일 컨셉**이나, 이번 지시로 수집 항목이 4개→6개로 구체화됨.
- **신규 내용**: (1) 2단계 에이전트 분리(수집 에이전트 vs 서비스완성 에이전트) 명시, (2) 관리자 검수 게이트, (3) 고객 게시판 경로로 결과 전달. 이 중 관리자 검수는 2차 문서의 "완전 자동화(AI 자동예약→자동발송)" 방향에 인간 확인 단계를 추가하는 변경이라 별도 확인이 필요하다.

## 1. 수집 에이전트 — 질의 항목 (6종, 확정)

1. 원하는 지역
2. 몇 박 며칠
3. 라운딩은 며칠 동안 (숙박일수와 별도 항목)
4. 인원수
5. 원하는 티오프 시간 (없을 시 "랜덤" 또는 "모르겠다" 응답 허용)
6. 항공권 예약 여부

수집 완료 후 예약신청서 양식으로 정리하여 고객이 신청서를 확인/수령할 수 있게 한다.

## 3. 2단계 에이전트 구조 (확정)

```
[Agent 1: 수집 + 인증]
  손님 대화 → 6개 항목 수집 → 예약신청서 작성 → 신원인증(4종 중 1) → 신청 확정
       ↓ (신청서 + 인증된 고객 데이터)
[Agent 2: 서비스 완성]
  락쿠텐 / 고라(GORA) / PGM 골프예약 사이트 정보를 확인해
  고객 요청(지역·일정·인원·시간·항공권 여부)에 맞게 세팅 → 제공 정보 완성
       ↓
[관리자 검수]  ← 예외 없이 매 건 필수 (2차 문서의 "완전자동" 전제를 대체)
  Agent 2 산출물을 관리자가 확인
       ↓ (승인)
[관리자 뷰]
  관리자가 볼 수 있는 고객 데이터 화면에 확정 정보 반영
  (별도 고객 게시판 불필요 — 이번 범위 아님)
```

## 4. 확정 내역 (2026-07-04 사용자 답변)

1. **관리자 검수 기준** — 고액·이상거래 구분 없이 **모든 건 공통 검수**. 3차 문서의 "결제 완료 시 자동 진행" 전제는 이 문서 기준으로 대체됨(승인 전까지 고객에게 최종 결과 미노출).
2. **고객 게시판** — 불필요. 이번 범위는 **관리자가 볼 수 있는 고객 데이터 화면**만 있으면 됨. 고객에게 결과를 되돌려주는 별도 전달 채널(마이페이지/알림톡 등)은 이번 지시에 포함되지 않음 — 필요 시 별도 확인.
3. **Agent 2 "서비스 완성"의 범위** — 락쿠텐·고라·PGM 3개 골프예약 사이트 정보를 확인해 고객 요청에 맞게 세팅한 정보 제공까지. 2차 문서의 "락쿠텐 고라 검색+가격 제시"에 **PGM이 신규 추가**됨.

### 크롤링 방식 채택에 따른 미확정 리스크 (착수 전 확인 필요)

크롤링 착수 전 아래 항목은 실제 대상 사이트별로 검증이 필요하다 — 코드 작성 전 필수 확인 대상이며, 이 문서만으로 바로 스크래퍼를 구현하지 않는다.

- **대상 사이트 정확한 URL/도메인 미확정** — "락쿠텐 GORA(gora.golf.rakuten.co.jp 추정)", "PGM(Pacific Golf Management 추정)"은 이전 대화에서 언급된 이름 기반 추정이며 실제 예약 시스템 URL·API 엔드포인트는 사용자 확인 또는 별도 조사가 필요하다.
- **이용약관(ToS) 상 크롤링 금지 여부** — 대형 예약 플랫폼은 대부분 자동 접근/스크래핑을 약관으로 금지한다. 재판매·중개 목적의 실시간 가격·잔여석 수집은 ToS 위반 및 국내외 법적 리스크(부정경쟁방지법, 저작권/DB권, 컴퓨터 부정접근 관련)로 이어질 수 있어 **법률 검토가 크롤링 구현보다 선행되어야 한다.**
- **robots.txt / rate-limit / anti-bot(캡차) 대응 범위** — 세 사이트 모두 실제 구조를 조사하기 전까지는 기술적 난이도를 알 수 없다.
- 위 리스크 확인 없이 크롤러를 프로덕션에 반영하지 않는다 (어뷰징 금지 원칙과 동일 선상).

## 7. 대상 사이트 구조 조사 (2026-07-04, 사용자 제공 URL 기준)

사용자 제공 3개 URL을 robots.txt·실제 예약 경로 기준으로 1차 조사했다. **이용약관(ToS) 원문은 아직 확보하지 못했다** — 아래 참고.

| 사이트 | 사용자 제공 URL | 실제 예약 시스템 도메인 | robots.txt 결론 |
|---|---|---|---|
| 楽天GORA | gora.golf.rakuten.co.jp/smart/iphone_top.html | 동일 도메인 (모바일 UI) | `/js/`, `/doc/rbs·test·ad/`만 차단. 검색 페이지 자체는 비차단 |
| Accordia Golf (KR) | accordiagolf.com/kr/ | **reserve.accordiagolf.com** (실제 예약) | `/global/` 전체 차단 + 특정 골프장 gid 다수 차단. KR 예약 경로가 `/global/`에 걸리는지 별도 확인 필요 |
| PGM (Pacific Golf) | pacificgolf.co.jp/index.asp | **booking.pacificgolf.co.jp** (실제 예약) | booking 서브도메인은 `Disallow:` 공백 — 전면 허용 |

**중요 정정**: 사용자가 준 메인 URL 3개는 전부 안내/포털 페이지였고, 실제 예약 검색·가격 조회는 별도 서브도메인(reserve.accordiagolf.com, booking.pacificgolf.co.jp)에서 일어난다. 크롤러가 실제로 붙어야 할 대상은 이 서브도메인들이다.

**ToS 미확보 사유**: `/company/rule/`, `/kr/privacy/`, `/kiyaku/` 등 통상적인 경로를 추정 접근했으나 전부 404. 각 사이트가 이용약관을 어디에 두는지 메인 페이지 하단 링크에서 직접 못 찾았다 (GORA는 콘텐츠 절단으로 확인 불가, Accordia KR·PGM은 하단에 약관 링크 자체가 안 보임 — 일본어 메인 사이트에 있을 가능성).

**robots.txt 허용 ≠ 크롤링 허용 아님** — robots.txt는 검색엔진 색인 정책일 뿐, 상업적 재판매 목적의 실시간 가격·잔여석 수집 허용을 의미하지 않는다. 셋 다 예약 가능 여부·가격은 지역/날짜 검색폼 제출 후 나오는 동적 결과일 가능성이 높아, 단순 정적 크롤러가 아니라 브라우저 자동화(세션 유지·폼 제출)가 필요할 것으로 보이며, 이는 "자동화 접속(bot) 금지" 조항이 있는 약관에서는 robots.txt 준수 여부와 무관하게 위반 소지가 있다.

### 다음 행동 (착수 전 필수)

1. **ToS 원문 확보** — 세 사이트 각각 실제 하단 푸터에서 이용약관/利用規約 링크를 직접 열어 크롤링·자동수집·재판매 금지 조항 유무 확인 (URL 추정 방식은 404만 반복돼 비효율적, 직접 방문 캡처가 더 빠름)
